# SmartLab 论文写作材料（答辩版）

一、设计选题的背景与意义、理论与实证准备、拟解决的问题、研究（设计）方法与技术路线

1. 设计选题的背景与意义

1.1 选题背景

本课题来自高校实验室在日常运行中的复合型安全管理需求。实验室管理工作并不只是查看单一指标，而是要同时关注环境参数、电气设备状态、水路状态、门窗安防状态、危化品状态和报警处理进度。上述对象在业务上互相影响，所以管理者在实际工作中需要“全局视角”和“快速联动”两种能力。传统管理方式通常依赖人工巡检、分散记录和阶段性汇总，这种模式在信息同步速度、异常联动效率和问题追踪完整性上都存在明显短板。SmartLab 项目围绕这一现实问题立项，并在系统目标上明确强调“统一感知、统一展示、统一预警、统一处置”，因此本课题具有清晰的问题来源和直接的应用场景。

本课题还具备扎实的工程背景，而不是概念化命题。项目文档已给出后端服务与数据服务配置，`AGENTS.md` 明确了 API 服务、PostgreSQL 和 MQTT 的地址与端口，`DATABASE.md` 明确了核心数据表结构和数据交互路径，`docs/USER_PERMISSION_DESIGN.md` 明确了角色与权限边界，`SmartLabApp.md` 给出了系统级方案。代码层面，项目主工程 `smart_lab` 已经形成完整目录结构，`core` 负责公共能力，`features` 负责业务模块，`shared/widgets` 负责复用组件，系统入口、依赖注入、路由配置、服务封装和页面骨架都已经落地。这些事实说明本课题已经处于“可实现、可验证、可答辩”的阶段。

1.2 选题意义

本课题首先具有直接的安全管理价值。系统把仪表盘、环境监测、电源管理、安防水路、危化品管理和报警中心六类业务放到一个统一入口，管理者不需要在多个平台之间切换，就可以在同一流程中完成“发现异常—定位来源—判断等级—执行处置”。这种能力会直接提升实验室安全管理的时效性和连续性。项目在实体层已经定义了 `Alert`、`SensorData` 等核心对象，在阈值层已经定义了风险分级规则，在页面层已经具备多模块状态展示能力，因此系统具备把“数据变化”转化为“管理动作”的基础条件。

本课题还具有明显的治理价值。系统不是单纯做看板，而是把角色权限和业务流程放在同一架构中。项目在用户实体中给出了角色能力判断，角色包括 `admin`、`teacher`、`graduate`、`undergraduate`，并实现了实验室访问能力、设备控制能力、报警确认能力和危化品操作能力判断。系统通过权限边界约束关键操作，并通过数据记录支撑后续追踪，这种设计能把实验室管理从“经验驱动”转向“规则驱动”。

本课题还具有工程实践和教学展示价值。项目技术栈覆盖移动端系统的关键能力面，包含 Flutter、BLoC、GoRouter、Dio、MQTT、Hive 和安全存储。系统在代码结构上已经具备较高可读性和模块化程度，适合用于毕业答辩场景中的结构化讲述。答辩时可以完整展示“需求建模—架构设计—模块实现—场景验证”的全过程，因此该课题不仅有应用价值，也有方法价值。

1.3 选题与项目场景的对应关系

本课题与项目现有场景是严格对应的。项目在 `lab_config.dart` 中配置了两个实验室场景，分别是“院楼806”和“西学楼新信科实验室”，并且给出了两套设备配置。院楼806配置了环境传感器、电源监测、智能插座、门磁和窗磁；西学楼新信科实验室在此基础上增加了水浸相关设备。系统在配置层已经具备“同架构管理多场景”的能力，论文可以据此开展场景对比分析。项目在 `mock_data_provider.dart` 中进一步体现了场景差异逻辑，例如水浸能力仅在对应场景启用，这为后续答辩演示中的场景切换提供了直接支撑。

2. 理论与实证准备

2.1 理论准备

2.1.1 架构理论准备

项目在架构层采用分层设计思想，文档和代码都体现了这一点。文档侧给出了 Clean Architecture 的目标结构，代码侧将公共能力集中在 `core`，将业务能力集中在 `features`，并在功能模块中区分领域对象和界面状态。该结构使系统具备较好的职责隔离能力，也为论文中的“可维护性”“可扩展性”“低耦合”分析提供了清晰依据。对答辩来说，这种结构非常重要，因为评审通常会先看系统是否有稳定架构，再看功能细节。

2.1.2 状态管理与数据流理论准备

项目在状态管理上全面采用 BLoC。认证、仪表盘、环境、电源、安防、危化品和报警模块都建立了对应的 Event、State 和 Bloc。该模式把“用户动作”和“系统状态”分离，能清楚描述状态变化过程，适合实验室这种多数据源和多状态并发的业务。项目实体层已经定义 `User`、`Lab`、`Device`、`SensorData`、`Alert`、`Chemical` 等核心对象，这些实体成为各模块状态建模的基础。论文在这一部分可以从“状态驱动界面”的角度解释系统可控性和可追踪性。

2.1.3 通信与存储理论准备

项目通信层采用 REST API 与 MQTT 双通道协作。API 负责认证、管理和历史数据查询，MQTT 负责遥测和告警实时推送。项目在 `mqtt_topics.dart` 中定义了统一主题规范 `lab/{buildingId}/{roomId}/{deviceType}/{deviceId}/{messageType}`，可以覆盖多实验室和多设备场景。项目存储层采用 Hive 与 Flutter Secure Storage 的组合，前者用于缓存和离线数据，后者用于令牌等敏感信息。该设计体现了“实时链路 + 管理链路 + 本地容灾”的系统思路，论文可以据此组织系统稳定性与安全性分析。

2.1.4 安全与权限理论准备

项目在安全治理方面已经形成基础规则。认证层实现了令牌处理与拦截器注入，权限层实现了角色能力判断，阈值层实现了指标到等级的映射，通知层实现了分级通道。`safety_thresholds.dart` 已定义温度、湿度、VOC、漏电流等阈值规则，例如温度正常区间 20–25°C、湿度正常区间 30–50%、VOC 正常上限 150、漏电流预警阈值 15mA、漏电流严重阈值 30mA。该规则体系使系统具备“从数据监测到风险分级再到预警处置”的逻辑闭环。

2.2 实证准备

2.2.1 文档与规范基础

项目文档体系已经覆盖研究和答辩需要的主要信息。`AGENTS.md` 给出项目范围、技术栈、服务配置和开发流程；`DATABASE.md` 给出数据库结构、缓存策略、同步机制和接口映射；`docs/USER_PERMISSION_DESIGN.md` 给出角色矩阵和页面流程；`SmartLabApp.md` 给出系统方案说明。项目还提供四类规范文档，分别对应 Flutter/Dart、BLoC 架构、UI 设计、后端与数据库。这些文档可以作为论文中设计依据与实现依据的引用来源，并且有助于答辩中的证据呈现。

2.2.2 代码与模块基础

项目主代码位于 `smart_lab/lib`，当前包含 58 个 Dart 文件。系统入口在 `main.dart`，初始化流程包含 Hive 初始化、依赖注入初始化和全局 BLoC 注入。依赖注入在 `injection.dart` 中使用 `get_it` 配置，认证 BLoC 使用全局单例，其余业务 BLoC 按工厂方式创建。`core/services` 提供 API、MQTT、通知和本地存储能力，`features` 提供业务能力，`shared/widgets` 提供复用组件。该结构已经具备论文“系统实现章节”的代码基础，也具备答辩演示中的模块拆解基础。

2.2.3 页面与交互基础

项目页面层已经形成连续演示路径。用户可以从登录页进入实验室选择页，再进入主框架页，主框架页提供仪表盘、环境、电源、安防、危化品入口，报警中心可以独立访问。页面层和状态层已经建立绑定关系，模块页面可以响应状态变化并反馈交互结果。项目在页面中已经实现多种关键交互，包括数据刷新、过滤筛选、状态切换、控制触发和告警确认。论文在这一部分可以用“页面职责—状态来源—交互动作—结果反馈”结构进行完整叙述。

2.2.4 数据链路与场景基础

项目已经具备完整的数据链路框架。API 服务封装了管理类接口，MQTT 服务封装了实时消息订阅与命令发布，实体层承担数据结构映射，BLoC 层承担状态聚合与转发，页面层承担最终展示与操作。项目同时保留了模拟数据能力用于稳定演示，这使系统在答辩阶段既具备业务可讲述性，也具备演示可控性。论文可以在这一部分重点说明系统如何实现“多源数据进入—统一状态管理—多模块同步展示”。

3. 拟解决的问题

3.1 业务目标与系统表达统一问题

本研究首要解决的问题是把业务目标转化为可验证的系统表达。项目当前已经覆盖多个业务模块，但论文需要进一步将这些模块组织成统一业务链路，明确每个模块的输入、处理和输出，以及各模块在安全管理闭环中的角色。该问题解决后，系统叙述会从“页面清单”升级为“业务系统”。

3.2 多源数据协同与状态一致性问题

本研究需要解决 API 数据、MQTT 数据和本地缓存在同一状态流中的协同问题。研究将明确数据来源职责、状态更新触发条件和跨模块共享边界，使数据链路表达更稳定，避免同一业务在不同页面出现语义不一致。该问题解决后，系统可解释性和可维护性都会提升。

3.3 报警闭环与处置机制标准化问题

本研究需要解决报警流程标准化问题。研究将把报警流程拆解为触发、分级、推送、确认和记录五个阶段，并结合现有实体与页面能力形成统一流程说明。该问题解决后，系统可以更清晰地体现“从异常发现到闭环处理”的全过程。

3.4 权限边界与管理执行一致性问题

本研究需要解决权限规则在页面层、状态层和服务层的一致执行问题。研究将围绕角色能力、实验室访问范围和关键操作约束建立统一权限语义，使系统管理边界更清楚，风险控制逻辑更稳定。该问题解决后，系统在治理能力上的说服力会明显增强。

3.5 答辩导向的成果组织问题

本研究还需要解决成果呈现组织问题。毕业答辩不仅关注功能是否存在，还关注研究链路是否完整、证据是否连续、结论是否可复述。研究将把文档依据、代码落点、场景验证和结论分析组织成统一证据链，使每个结论都能在系统中找到对应支撑。

4. 研究（设计）方法与技术路线

4.1 研究（设计）方法

4.1.1 文档对照分析法

研究先采用文档对照分析法建立事实基线。研究将项目文档和规范文档与代码实现逐项映射，形成“设计依据—实现位置—验证路径”的对应关系，以保证论文内容与项目事实一致。

4.1.2 架构映射拆解法

研究采用架构映射拆解法分析系统结构。研究会从 `core` 到 `features`，从 `services` 到 `bloc`，从 `entities` 到 `pages` 逐层拆解，明确各层职责和依赖关系。该方法有助于在答辩中清楚解释系统为什么这样设计。

4.1.3 场景驱动迭代法

研究采用场景驱动迭代法推进设计与实现。研究不按单页面推进，而按完整业务场景推进，每次迭代都覆盖进入系统、查看状态、执行操作、观察反馈的全过程。该方法可以保证系统演示路径连续，也能让论文验证章节更有实证性。

4.1.4 结果归纳验证法

研究采用结果归纳验证法形成结论。研究将对关键场景记录输入条件、系统响应和输出结果，再按模块能力、链路稳定性和治理可执行性进行归纳。该方法可以把实现过程转换为结构化证据，支撑答辩论证。

4.2 技术路线

4.2.1 路线总览

本研究采用“基线整理—结构对齐—链路联调—场景验证—成果固化”的五段式路线。每个阶段同步产出代码成果与文档成果，确保实现进度和论文进度一致。

4.2.2 基线整理阶段

该阶段聚焦边界确认，任务包括模块边界、接口边界、角色边界和场景边界确认，输出系统全景图和模块关系图。

4.2.3 结构对齐阶段

该阶段聚焦关键链路一致性，任务包括认证链路、路由链路、状态链路和页面链路对齐，确保业务语义统一。

4.2.4 链路联调阶段

该阶段聚焦模块协同，任务包括打通仪表盘、环境、电源、安防、危化品和报警中心之间的数据路径，明确 API 与 MQTT 的职责分工。

4.2.5 场景验证阶段

该阶段聚焦可验证证据，任务包括按典型安全场景执行验证并形成标准记录，重点验证状态变化、预警触发、处置动作和记录沉淀。

4.2.6 成果固化阶段

该阶段聚焦答辩表达，任务包括将设计依据、实现路径、验证结果和结论分析统一固化到论文与答辩材料中，形成可演示、可复述、可追溯的成果包。

4.3 预期成果与答辩展示路径

本研究预期形成系统成果、研究成果和表达成果三类产出。系统成果体现为多模块联动的智慧实验室移动端应用，研究成果体现为完整的设计与实现文档，表达成果体现为结构化答辩讲述和可复现实验演示。答辩展示建议采用“登录与权限进入—实验室选择—仪表盘总览—模块深入—报警闭环”的路径，这条路径与项目当前页面结构和状态流高度一致，能够在有限时间内完整体现课题价值。

三、工作步骤与时间安排

1. 工作步骤

1.1 第一阶段：研究准备与基线建立

1.1.1 本阶段将完成项目文档、代码结构、业务模块和数据路径的系统化梳理，形成统一的研究范围说明和系统基线说明，为后续实现与论文写作提供稳定框架。  
1.1.2 本阶段将同步建立答辩讲述主线，提前确定“业务问题—技术方案—实现路径—验证结论”的演示顺序，确保后续材料和答辩逻辑始终一致。

1.2 第二阶段：系统设计细化与关键链路完善

1.2.1 本阶段将聚焦认证链路、实验室切换链路、仪表盘聚合链路以及报警处理链路，进一步细化模块职责和交互关系，使系统主流程具备完整叙述能力。  
1.2.2 本阶段将完善数据接口描述、消息主题描述和状态流描述，确保论文中的设计图、流程图和代码实现能够一一对应。

1.3 第三阶段：模块联调与场景化验证

1.3.1 本阶段将围绕环境、电源、安防水路、危化品、报警中心等核心模块开展联调，重点验证模块间的数据一致性和状态传递一致性。  
1.3.2 本阶段将按典型实验室场景组织验证过程，并形成“输入条件—系统响应—结果判定”的记录模板，为论文测试章节提供直接材料。

1.4 第四阶段：论文撰写与材料整理

1.4.1 本阶段将按照既定提纲完成论文初稿与二稿，重点保证章节间逻辑连续、术语一致、图文对应。  
1.4.2 本阶段将同步整理系统演示截图、流程图、模块关系图和测试记录表，确保答辩材料和论文内容在结构上完全对齐。

1.5 第五阶段：答辩准备与表达优化

1.5.1 本阶段将完成答辩 PPT、演示脚本和问答清单，并围绕评审常见问题准备标准化回答。  
1.5.2 本阶段将通过多轮演练优化表达节奏，确保答辩现场能够清晰说明研究价值、技术路径和实现结果。
