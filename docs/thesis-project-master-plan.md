## 🎓 答辩常见问题预备

### Q1: 你为什么选择 BLoC 而不是其他状态管理模式？

**预备答案**：
> 在项目的场景中，我们需要处理多个数据源（REST API、MQTT、本地缓存）的事件，以及7个业务模块的状态同步。相比 Redux（模板代码多）和 Riverpod（学习曲线陡），BLoC 在事件驱动和 Stream 聚合上更清晰。具体地，当用户切换实验室时，AuthBloc 产生 LabChanged 事件，这个事件被所有模块的 BLoC 订阅，触发各自的数据重新加载。这种事件传播机制在 BLoC 中实现最直观。

### Q2: 权限管理为什么采用"三层检查"（页面/状态/服务）？不会重复吗？

**预备答案**：
> 是的，看起来有重复，但这是防御式设计：
> - 页面层检查是**用户体验**（禁用按钮，避免误操作）
> - 状态层检查是**业务一致性**（防止BLoC处理不该发生的事件）
> - 服务层检查是**安全防线**（防止客户端被破解后越权调用API）
> 
> 三层检查虽然代码有重复，但确保了没有单点失败，这在安全系统中是必要的。

### Q3: 为什么 MQTT 消息要本地缓存？REST API 不够吗？

**预备答案**：
> REST API 是"拉取"模式（客户端主动查询），而 MQTT 是"推送"模式（服务器主动发送）。在实时监测系统中，报警是由服务器触发的，不能依赖客户端定时拉取（那样会有延迟）。但MQTT 消息可能因网络丢失而漏掉。所以我们在客户端本地缓存 MQTT 消息（存到 Hive），当网络恢复时补偿处理。这样既保证了实时性，又保证了可靠性。
